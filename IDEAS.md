## 🎮 מצבי משחק חדשים

### רעיון #1 — "המקום החבוי" · ציד אוצר יומי רב-משתתפים

**סטטוס:** רעיון גולמי · **תאריך:** 2026-06 · **מקור:** רותם

#### הקונספט
אתגר **יומי** עם **מספר משתתפים בלתי מוגבל**. כל יום מתפרסמת תמונה בודדת
של **מקום פיזי אמיתי** בעולם האמיתי, מלווה ב**רמזים** (טקסט/ויזואל). במקום
המסתור הוסתר מראש **פתק עם קוד / ברקוד**. המשתתף:
1. צופה בתמונה וברמזים באפליקציה (אי אפשר לנחש מתוך האפליקציה — רק לזהות).
2. מזהה את המקום הפיזי ומגיע אליו **בפועל**.
3. מוצא את הפתק המוסתר וסורק את הברקוד / מקליד את הקוד באפליקציה.
4. מקבל **שובר לפרס** (קופון).

זהו מעבר ממשחק "על המסך" למשחק שמוציא את השחקן **לעולם האמיתי** — שילוב
של geocaching + חידת זיהוי מקומות + פרסים אמיתיים.

#### זרימת המשתמש (User Journey)
1. **מסך אתגר יומי** — תמונה + רמזים מדורגים (רמז ראשון חינם, רמזים נוספים
   בעלות מטבעות, בדומה למנגנון הרמזים הקיים במשחק).
2. **שלב הזיהוי** — המשתתף מקליד/בוחר את שם המקום. אם צדק → נחשפת כתובת/אזור
   כללי (לא מדויק מדי, כדי לשמור על חוויית החיפוש).
3. **שלב הפיזי** — המשתתף מגיע למקום, מחפש את הפתק.
4. **שלב הגאולה** — סריקת ברקוד / הקלדת קוד → אימות → שובר פרס נכנס לארנק.
5. **לוח תוצאות יומי** — מי מצא ראשון, כמה מצאו היום, רצף ימים (streak).

#### שיקולים טכניים
- **מניעת רמאות (קריטי):** הסיכון המרכזי — מי שמוצא את הפתק מצלם/משתף את הקוד
  ברשת, ואחרים גואלים בלי להגיע פיזית. מנגנוני הגנה אפשריים:
  - **אימות GPS** — הגאולה מותרת רק כש-`location` של המכשיר בטווח X מטר
    מנקודת המסתור (קואורדינטות נשמרות בצד השרת, לא נחשפות).
  - **קוד חד-פעמי לכל פתק** + מספר פתקים פיזיים מפוזרים (כל אחד קוד שונה).
  - **חלון זמן** — קוד תקף רק לאותו יום.
  - **קצב/anti-abuse** — הגבלת ניסיונות גאולה, זיהוי חשבונות מרובים.
  - שילוב: GPS-in-range **וגם** קוד תקף = הגאולה היחידה שמתקבלת.
- **סריקת ברקוד:** חבילה כמו `mobile_scanner` (Flutter). הקוד יכול להיות
  QR שמכיל מזהה ייחודי + חתימה.
- **מודל פרסים/מונטיזציה:** עסקים מקומיים יכולים **לממן** את המקום והפרס
  (קופון להטבה אצלם) → ערוץ הכנסה + תנועה לעסק. שוברים נשמרים ב-Firestore עם
  קוד מימוש חד-פעמי.
- **סקיילינג ל"בלתי מוגבל":** מאחר שאין הגבלת משתתפים, המודל הוא **מרוץ** או
  **"כל מי שמצא — זוכה"** (לא חדר עם תורות). אין צורך במנוע התורות/החדרים
  הקיים — זהו זרם נתונים גלובלי יומי + לוח תוצאות.
- **איפוס יומי** (UTC, עקבי עם הפרס היומי הקיים) + מקום/תמונה חדשים בכל יום.
- **מבנה נתונים מוצע (Firestore):**
  - `dailyHunts/{yyyy-mm-dd}` — `imageUrl`, `hints[]`, `answer`,
    `geo` (lat/lng, server-only), `radiusM`, `prize`, `sponsor`.
  - `dailyHunts/{date}/redemptions/{uid}` — `redeemedAt`, `code`, `voucherId`.
  - `vouchers/{voucherId}` — `uid`, `prize`, `redemptionCode`, `status`.

#### שאלות פתוחות
- האם הפרס לכולם שמצאו, או רק ל-N הראשונים?
- כיסוי גאוגרפי — מתחילים בעיר/אזור אחד או כל הארץ?
- מי מסתיר את הפתקים פיזית בשלב הראשון (צוות / שותפים / משתמשי-על)?
- גיבוי אם הפתק נגנב/אבד — קוד דיגיטלי חלופי באתר/GPS בלבד?
- פרטיות מיקום — בקשת הרשאת `location` רק בזמן הגאולה, עם הסבר ברור.

#### תלות בקוד הקיים (re-use)
- מנגנון **הרמזים המדורג** (40/80 מטבעות) — כבר קיים, ניתן לעיבוד מחדש.
- **הפרס היומי + streak** (איפוס UTC) — תבנית קיימת.
- **ארנק/שוברים** — להרחיב את `economy_service` עם סוג טרנזקציה `voucher`.
- **זיהוי מקומות** — מאגר התמונות והתשובות הקיים (`game_image_model`).
